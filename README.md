# AI 校规问答助手：基于 RAG 检索与本地大模型裁判的校园规则问答系统

## 1. 项目简介

本项目是一个面向校园规则查询场景的 AI 问答系统。系统可以读取本地校规文本，根据用户的自然语言问题检索相关条款，并输出“允许 / 不允许 / 有条件允许 / 未明确”的结构化判定结果。

与普通聊天机器人不同，本项目并不是直接让大模型自由回答，而是采用 RAG（Retrieval-Augmented Generation，检索增强生成）思路：先从校规知识库中检索相关条款，再基于检索结果进行判断，从而减少大模型胡编、误答和引用无关规则的问题。

项目支持两种裁判方式：

- 本地规则引擎裁判：根据关键词、行为场景和规则类型进行判定；
- Ollama 本地大模型裁判：调用本地部署的大语言模型，对候选校规进行最终判断。

同时，项目提供了命令行版本和 Streamlit 可视化版本，方便测试、演示和后续扩展。

---

## 2. 项目背景

学生在查询校规时，往往不会使用正式条款中的表达，而是会用非常口语化的问题，例如：

- 上课可以喝水吗？
- 在食堂吃饭算违规吗？
- 考试作弊会怎么样？
- 请病假需要什么手续？
- 上课睡觉可以吗？

如果只使用关键词搜索，系统容易因为表达不一致而漏掉相关条款；如果直接使用大模型回答，又可能出现编造校规、引用错误、回答过度自信等问题。

因此，本项目尝试构建一个更可靠的校园规则问答流程：

```text
用户问题
  ↓
查询扩展
  ↓
向量检索
  ↓
关键词重排
  ↓
候选校规筛选
  ↓
本地规则引擎 / Ollama 大模型裁判
  ↓
结构化判定结果
```

## 3. 核心功能

### 3.1 校规文本解析

系统从 data/school.txt 中读取校规文本，并按照如下格式进行条款切分：

1. 学生应当遵守课堂纪律，认真听讲。
2. 严禁考试作弊、替考等违反考试纪律的行为。
3. 学生请假应提前办理相关手续。

每一条校规会被解析成独立文本片段，后续用于向量化和检索。

### 3.2 中文语义向量检索

项目使用 SentenceTransformer 加载中文 embedding 模型，将校规条款转换为向量。

默认 embedding 模型为：**BAAI/bge-small-zh**

当用户输入问题后，系统会将问题同样转换为向量，并通过向量相似度召回候选校规。

### 3.3 查询扩展

为了提升口语化问题的检索命中率，系统设计了查询扩展机制。

例如：

```text
翘课 → 无故缺课 旷课 缺席 课程
打架 → 斗殴 动武 身体伤害 校园欺凌
吃饭 → 用餐 吃零食 餐具 食堂 教室
考试作弊 → 作弊 交头接耳 传递纸条 替考
```

这样可以缓解“用户口语表达”和“校规正式表达”之间的不一致问题。

### 3.4 关键词重排

纯向量检索有时会命中字面相似但实际场景不同的条款。为了解决这个问题，系统加入了关键词重排机制。

重排时会重点考虑：

```text
行为词：如缺课、迟到、斗殴、作弊、请假、吸烟等；
场景词：如教室、课堂、食堂、宿舍、图书馆等；
规则词：如严禁、不得、禁止、必须、应当、可以等；
用户问题中的局部字面匹配。
```

最终排序结果由语义相似度和关键词加分共同决定。

### 3.5 本地规则引擎判定

系统内置本地规则引擎，可以在不调用大模型的情况下完成基础判定。

本地规则引擎会输出四类结果：**允许；不允许；有条件允许；未明确**

本地规则引擎重点解决以下问题：

禁止类规则必须匹配行为和场景；

字面相似但行为不同的规则不能直接引用；

没有明确禁止时，不直接判定为“不允许”；

对高频易误判问题进行特殊处理。

```text
例如：

用户问：上课可以喝水吗？
系统不会因为“饮水机不得洗脸、洗衣物”等条款误判为不允许喝水。

再如：

用户问：在食堂吃饭可以吗？
系统不会因为“教室、实验室、图书馆禁止用餐”等条款误判为食堂不能吃饭。
```

### 3.6 Ollama 本地大模型裁判

项目支持通过 Ollama 调用本地大语言模型进行最终裁判。

默认模型名称为：**deepseek-r1:8b**

大模型裁判并不是直接自由回答，而是只能根据系统检索到的候选校规进行判断。系统提示词会约束模型：

```text
只能依据候选校规回答；
不允许编造校规；
字面相似但行为不同的条款不能引用；
必须输出固定格式；
不输出多余思考过程。
```

输出格式如下：

```text
判定：允许/不允许/有条件允许/未明确
原因：一句话说明依据
来源：第X条校规/未检索到明确相关条款
```

如果 Ollama 调用失败，系统会自动回退到本地规则引擎，避免整个系统不可用。

### 3.7 Streamlit 可视化界面

项目提供 Streamlit 前端页面，支持：

```text
聊天式提问；
显示结构化判定结果；
显示判定来源；
展示检索到的相关校规；
调试模式查看检索分数；
调整 Top-K 检索数量；
开启或关闭 Ollama 裁判；
清空对话记录；
重载系统缓存。
```

## 8. 运行方式

### 8.1 命令行版本

运行：

```text
<>Bash
python main.py
```

运行后可以在终端中输入问题：

```text
请输入问题：上课可以喝水吗？
```

系统会输出：

```text
判定：允许
原因：校规中没有明确禁止上课喝水。一般可以喝水，但不应影响课堂秩序。
来源：未检索到明确禁止条款
裁判：本地规则引擎
```

关闭 Ollama，只使用本地规则引擎：

```text
<>Bash
python main.py --no-ollama
```

开启调试模式：

```text
<>Bash
python main.py --debug
```

自定义校规路径：

```text
<>Bash
python main.py --rule-path data/school.txt
```

自定义模型名称：

```text
<>Bash
python main.py --model-name deepseek-r1:8b
```

### 8.2 Streamlit 可视化版本

运行：

```text
<>Bash
streamlit run app.py
```

打开浏览器后，可以在页面中输入问题并查看结果。

页面左侧可以配置：

```text
校规文件路径；
Embedding 模型；
检索候选数量 Top-K；
最大上下文字符数；
是否启用 Ollama 本地大模型裁判；
Ollama 路径；
Ollama 模型名称；
是否开启调试模式。
```

## 10. 项目亮点

### 10.1 不只是调用大模型

本项目不是简单地把问题丢给大模型，而是先检索校规，再基于候选校规进行判断，减少大模型幻觉问题。

### 10.2 具有可解释性

系统会展示检索到的相关校规，用户可以看到判定结果来自哪些条款。

在调试模式下，还可以查看：

```text
final_score
semantic_score
keyword_score
```

这有助于理解系统为什么召回某些校规。

### 10.3 针对中文校园场景做了优化

项目加入了中文口语表达扩展和场景过滤逻辑，可以处理一些容易误判的问题。

例如：

```text
“喝水”不等于“饮水机洗脸”
“食堂吃饭”不等于“教室禁止用餐”
“带出餐具”不等于“不能吃饭”
```

### 10.4 支持本地化部署

项目可以使用 Ollama 调用本地大模型，不依赖在线大模型 API。

如果本地大模型不可用，也可以关闭 Ollama，使用本地规则引擎完成基础判断。

### 10.5 适合作为 RAG 入门项目

本项目覆盖了一个基础 RAG 应用的核心流程：

- **文本加载 → 文本切分 → 向量化 → 语义检索 → 重排 → 生成/判定 → 结果展示**

适合用于理解企业知识库问答、智能客服、政策问答、规章制度查询等 AI 应用场景。

## 12. 当前局限

本项目仍然是一个学习型和演示型项目，存在以下不足：

```text
校规文本依赖固定编号格式，如果原始文档格式复杂，需要额外清洗；
当前没有接入真正的数据库或向量数据库，向量数据在程序启动时生成；
查询扩展词表仍然依赖人工维护；
本地规则引擎只能覆盖部分高频场景；
多轮对话目前主要用于页面展示，历史消息不会参与下一次检索；
如果使用 Ollama，本地模型质量和运行速度会影响回答效果；
目前没有用户权限系统、后台管理系统和日志分析模块。
```
